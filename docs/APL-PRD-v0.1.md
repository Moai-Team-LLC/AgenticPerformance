| | |
|---|---|
| **Документ** | Требования к разработке (SRS / PRD-гибрид) |
| **Продукт** | Agent Performance Layer (APL) — модуль AgenticMind |
| **Версия документа** | v0.1 (черновик к ревью) |

---

## 1. Назначение и контекст
Moai Team строит множество agentic/workflow-продуктов (собственных и клиентских), где часть процесса исполняется LLM-агентами. Сегодня каждый продукт наблюдается и улучшается вручную и по-своему. APL — переиспользуемый слой, который любой продукт подключает через тонкий SDK и который превращает сырое исполнение агентов в: (1) наблюдаемость и трейсинг, (2) оценку качества на golden set, (3) таксономию ошибок с трендами, (4) циклический процесс улучшения каждого агента индивидуально.

APL реализуется **как модуль существующего движка AgenticMind** (why-trace, auditable memory, judge-gated self-improvement), а не как система с нуля. Это одновременно внутренний инструмент, стандарт для всех продуктов Moai и монетизируемая enterprise-поверхность AgenticMind.

### 1.1. Цели (измеримые исходы)
- **G1.** Любой продукт с агентами подключается к APL за ≤ 1 день инженерных усилий (time-to-first-trace).
- **G2.** Для каждого агента существует версионируемый golden set и тренд качества во времени; регрессии не попадают в прод (блокируются CI-гейтом).
- **G3.** Ошибки каждого агента классифицированы в ≤ 10 именованных кластеров с трендами, а не наблюдаются как сырые трейсы.
- **G4.** Существует управляемый цикл улучшения агента с тремя уровнями автономии (ручной → Claude-in-loop → judge-gated авто), с жёсткими границами безопасности.
- **G5.** Один и тот же слой работает в SaaS- и self-hosted/VPC-режимах поверх мультитенантной изоляции.

### 1.2. Не-цели (явно вне охвата)
- **NG1.** APL **не** заменяет и не переписывает agentic-логику продуктов; он инструментирует их извне.
- **NG2.** APL **не** привязан к конкретному фреймворку (LangGraph/CrewAI/Claude SDK) и не навязывает его.
- **NG3.** APL **не** строит собственный UI-конструктор агентов и не выполняет их оркестрацию.
- **NG4.** В MVP APL **не** делает автономную дообучку/файн-тюнинг моделей; улучшение — на уровне промпта/контекста/инструментов/few-shot.
- **NG5.** APL **не** внедряет проприетарный формат телеметрии — базируется на OpenTelemetry.

## 2. Область охвата
**В охвате:** контракт телеметрии, SDK инструментации, приём и хранение трейсов, реестр агентов и версий, движок оценки (evals), судьи и их калибровка, анализ и кластеризация ошибок, цикл улучшения (L1–L3), скоркарта/контрол-плейн, адаптеры под фреймворки и режимы деплоя.

**Вне охвата:** см. Не-цели (§1.2). Также вне MVP: A/B-инфраструктура промоушена (переносится в фазу 5), мультирегиональный деплой, готовые вертикальные пакеты evals.

---

## 3. Глоссарий
- **Агент** — сущность продукта, исполняющая задачу через LLM внутри детерминированного контура.
- **agent_version** — иммутабельный снапшот конфигурации агента (hash промпта, инструменты, модель, параметры, context-стратегия).
- **Трейс / спан** — запись исполнения по конвенциям OpenTelemetry GenAI.
- **Golden set** — курируемый+намайненный набор эталонных кейсов оценки для конкретного агента.
- **Judge (судья)** — LLM-оценщик с записью калибровки против человеческих меток (TPR/TNR).
- **Failure cluster** — именованная категория повторяющейся ошибки конкретного агента.
- **Why-trace** — записанные точки решения агента (почему выбран этот шаг), из AgenticMind.
- **Скоркарта** — материализованный вид «производительности» агента во времени.
- **Cycle of Trust** — принцип: границы прав и разрушительные действия форсируются кодом, не промптом.

---

## 4. Заинтересованные стороны и роли
- **Product Engineer** — интегрирует SDK, читает скоркарты, применяет патчи (L1/L2).
- **Platform/Ops** — эксплуатирует APL, задаёт границы автономии и политики авто-улучшения.
- **Client Tenant (enterprise)** — потребитель self-hosted/VPC-инсталляции, требует изоляции/аудита.
- **Claude-in-the-loop** — предлагает патчи из кластеров ошибок (L2), проверяется судьёй (L3).
- **Аудитор/комплаенс** — потребитель audit-логов и why-trace (позиционирование под EU AI Act).

---

## 5. Обзор решения (карта из 7 слоёв)
0. **Контракт** — OTel GenAI semconv + namespaced `moai.*` атрибуты.
1. **SDK** — тонкая обёртка agent loop + адаптеры фреймворков.
2. **Приём/хранение** — OTLP ingestion; Postgres+pgvector (состояние) + ClickHouse (трейсы).
3. **Реестр** — агенты и иммутабельные версии.
4. **Оценка** — трёхуровневый eval-пайплайн + CI-гейт.
5. **Анализ ошибок** — автотриаж + семантическая кластеризация + тренды.
6. **Улучшение** — L1 ручной → L2 Claude-in-loop → L3 judge-gated авто.
7. **Скоркарта/контрол-плейн** — per-agent review + fleet-вид.

---

## 6. Функциональные требования

### 6.1. Контракт телеметрии (FR-CONTRACT)
- **FR-CONTRACT-1 (MUST).** Формат телеметрии базируется на OpenTelemetry GenAI semantic conventions; собственный формат не вводится.
- **FR-CONTRACT-2 (MUST).** Дерево трейса использует канонические спаны: корневой `invoke_agent`, дочерние `chat` (вызовы LLM) и `execute_tool` (вызовы инструментов).
- **FR-CONTRACT-3 (MUST).** Захватываются стандартные атрибуты: `gen_ai.request.model`, `gen_ai.usage.input_tokens`/`output_tokens`, `gen_ai.response.finish_reasons`, `gen_ai.operation.name`, `gen_ai.provider.name`.
- **FR-CONTRACT-4 (MUST).** Экспериментальные agent/tool-спаны включаются через опт-ин (`OTEL_SEMCONV_STABILITY_OPT_IN=gen_ai_latest_experimental`); версия конвенций фиксируется в конфиге APL.
- **FR-CONTRACT-5 (MUST).** Поверх стандарта эмитится namespaced набор `moai.*`: `agent_id`, `agent_version`, `product_id`, `tenant_id`, `task_id`, `outcome`, `human_feedback`, `decision_reason`. Namespace обязателен для защиты от коллизий с будущими атрибутами OTel.
- **FR-CONTRACT-6 (MUST).** Trace context пробрасывается через асинхронные границы саб-агентов, чтобы единое дерево исполнения не рассыпалось в orphan-спаны (главный failure mode multi-agent наблюдаемости).
- **FR-CONTRACT-7 (SHOULD).** Контент-атрибуты (`gen_ai.input.messages`, `gen_ai.output.messages`, `gen_ai.system_instructions`) захватываются управляемо (флагом), с учётом их объёма и чувствительности.

**Критерии приёмки:** трейс агента с ≥1 tool-call, снятый из продукта на LangGraph и из продукта на голом коде, даёт идентичную структуру дерева и обязательный набор `gen_ai.*` + `moai.*` атрибутов; multi-agent сценарий отображается одним связным деревом.

### 6.2. SDK инструментации (FR-SDK)
- **FR-SDK-1 (MUST).** SDK на TypeScript; предоставляет декоратор/HOF-обёртку agent loop, автоматически открывающую `invoke_agent` и проставляющую `moai.*`.
- **FR-SDK-2 (MUST).** Автоинструментация вызовов инструментов (`execute_tool`) без ручного кода на каждый инструмент.
- **FR-SDK-3 (MUST).** Хуки на outcome (`success|fail|escalated|unknown`) и на human-feedback (thumbs/rubric/correction).
- **FR-SDK-4 (MUST).** Контроль сэмплинга: 100% на eval-трафике; настраиваемый tail-sampling на проде (по умолчанию ~5%, 100% на ошибках и медленных ходах).
- **FR-SDK-5 (MUST).** Редакция PII на эмите (redaction-процессор коллектора или Presidio-сайдкар) до экспорта.
- **FR-SDK-6 (SHOULD).** Python-SDK с паритетом ключевой поверхности для продуктов на Python.
- **FR-SDK-7 (SHOULD).** Захват why-trace (точек решения) через явный вызов SDK.
- **FR-SDK-8 (MAY).** Оверхед-бюджет: батч-экспорт, влияние на основной поток < 1%.

**Критерии приёмки:** интеграция в новый продукт (install → обернуть loop → задать `agent_id` → указать endpoint) занимает ≤ 1 день; трейсы появляются в хранилище без ручной инструментации каждого инструмента.

### 6.3. Приём и хранение (FR-INGEST)
- **FR-INGEST-1 (MUST).** Приём телеметрии по OTLP; совместимость с OTel Collector.
- **FR-INGEST-2 (MUST).** Состояние (реестр, golden sets, eval-раны, калибровки, скоркарты, эмбеддинги фейлов) хранится в Postgres+pgvector.
- **FR-INGEST-3 (MUST).** Поток трейсов на масштабе хранится в ClickHouse (append-heavy аналитика).
- **FR-INGEST-4 (MUST).** Ретеншн трейсов настраиваем и достаточен для расследования инцидентов (типовой диапазон 30–90 дней).
- **FR-INGEST-5 (SHOULD).** Семантический поиск по фейл-трейсам через pgvector.

### 6.4. Реестр агентов и версий (FR-REG)
- **FR-REG-1 (MUST).** Реестр `agent`: `id`, `product_id`, `task_description`, `owner`.
- **FR-REG-2 (MUST).** `agent_version` — иммутабельный снапшот: `prompt_hash`, `prompt`, `tools[]`, `model`, `params`, `context_strategy`, `git_ref`, `created_at`.
- **FR-REG-3 (MUST).** Каждый трейс, eval-ран и improvement привязаны к конкретной паре (`agent_id`, `agent_version`).
- **FR-REG-4 (MUST).** Смена любого элемента конфига порождает новую версию; редактирование существующей версии запрещено.

**Критерии приёмки:** по любому трейсу восстанавливается точная конфигурация агента; тренд качества строится по версиям; регрессия атрибутируется к конкретной версии.

### 6.5. Движок оценки (FR-EVAL)
- **FR-EVAL-1 (MUST).** Golden set определяется **на каждого агента отдельно**; единый глобальный сьют не используется.
- **FR-EVAL-2 (MUST).** Три уровня оценки: (a) детерминированные ассерты (валидность tool-call, схема ответа, бюджеты latency/cost); (b) откалиброванный LLM-judge (качество, faithfulness); (c) ротирующийся human-сэмпл как ground truth.
- **FR-EVAL-3 (MUST).** Golden set засевается вручную и растёт из прода (еженедельный сэмпл фейлов → разметка → добавление кейсов).
- **FR-EVAL-4 (MUST).** CI-гейт: eval-сьют прогоняется на каждую смену `agent_version` и блокирует деплой при регрессии относительно текущей прод-версии.
- **FR-EVAL-5 (SHOULD).** 100% продакшн-трафика формирует трейсы, пригодные для последующего пополнения eval-сета.

**Критерии приёмки:** попытка задеплоить версию с падением метрики на golden set блокируется автоматически; тренд score по агенту виден на скоркарте.

### 6.6. Судьи и калибровка (FR-JUDGE)
- **FR-JUDGE-1 (MUST).** Каждый judge — версионируемая сущность с промптом и записью калибровки.
- **FR-JUDGE-2 (MUST).** Калибровка требует ≥ 100 человеческих меток; TPR и TNR обе > 80%.
- **FR-JUDGE-3 (MUST).** При изменении промпта судьи калибровка перезапускается; TPR/TNR публикуются с каждым релизом.
- **FR-JUDGE-4 (SHOULD).** Human-аннотации (FR-EVAL-2c) переиспользуются для калибровки и для таксономии ошибок.

**Критерии приёмки:** судья без валидной калибровки (TPR/TNR ≥ порога) не допускается к гейтингу деплоя и к авто-улучшению.

### 6.7. Анализ ошибок (FR-FAIL)
- **FR-FAIL-1 (MUST).** Автотриаж: упавший ассерт, низкий score судьи, human thumbs-down и событие эскалации автоматически попадают в пайплайн ошибок.
- **FR-FAIL-2 (MUST).** Кластеризация: эмбеддинг фейл-трейсов (pgvector) → семантические кластеры → LLM open coding → axial coding в стабильные именованные категории на агента.
- **FR-FAIL-3 (MUST).** Trend-детекция по кластерам: алерт при всплеске кластера или появлении нового кластера после деплоя.
- **FR-FAIL-4 (MUST).** Cost/latency-телеметрия per agent / per version / per cluster.
- **FR-FAIL-5 (SHOULD).** Типовая цель: ошибки агента сводятся к 3–7 повторяющимся кластерам.

**Критерии приёмки:** для агента доступен список именованных кластеров с количеством и трендом и репрезентативными трейсами; после релиза, ухудшившего поведение, соответствующий кластер даёт алерт.

### 6.8. Цикл улучшения (FR-IMPROVE)
- **FR-IMPROVE-1 (MUST).** **L1 (Assisted):** скоркарта показывает топ-кластер и репрезентативные трейсы; инженер патчит и шипит через eval-гейт.
- **FR-IMPROVE-2 (MUST).** **L2 (Suggested):** по кластеру ошибок LLM предлагает конкретный патч (правка промпта, описание инструмента, context-стратегия, few-shot из хороших трейсов) с гипотезой; инженер ревьюит; шип через eval-гейт.
- **FR-IMPROVE-3 (SHOULD).** **L3 (Judge-gated авто):** propose patch → полный eval → независимый откалиброванный судья подтверждает прирост без регрессии на golden set и без новых кластеров → авто-мёрж в canary → A/B против прода → промоут или откат.
- **FR-IMPROVE-4 (MUST).** Границы автономии: авто-эволюция **только** промпта/контекста/few-shot. Инструменты с сайд-эффектами, права доступа и границы доверия авто-патчами **не изменяются никогда** (Cycle of Trust).
- **FR-IMPROVE-5 (MUST).** Каждое улучшение фиксируется как `improvement`: `version_from→version_to`, `hypothesis`, `eval_delta`, `status`, `author` (human/claude/judge-gated).
- **FR-IMPROVE-6 (MUST).** Оператор конфигурирует, какие типы патчей и для каких агентов допустимы к авто-применению (L3 — opt-in на агента).

**Критерии приёмки:** попытка авто-улучшения изменить набор инструментов или права отклоняется; журнал improvements даёт полную трассируемость «что изменилось → как повлияло на score».

### 6.9. Скоркарта и контрол-плейн (FR-SCORE)
- **FR-SCORE-1 (MUST).** Per-agent вид: текущая версия, кривая score во времени, топ-кластеры и их тренды, cost/latency/token бюджеты vs факт, % эскалации, % успешных tool-call, ожидающие апрува улучшения.
- **FR-SCORE-2 (MUST).** Diff между версиями: что изменилось и как это повлияло на метрику.
- **FR-SCORE-3 (SHOULD).** Fleet-вид поверх всех продуктов, ранжированный по здоровью агентов (для приоритизации инженерного времени).
- **FR-SCORE-4 (MAY).** Экспорт скоркарты/метрик во внешние бэкенды (Datadog/Grafana/Phoenix) через OTel-совместимый выход.

### 6.10. Интеграция, адаптеры, деплой (FR-INTEG)
- **FR-INTEG-1 (MUST).** Мультитенантность с первого дня по `product_id`/`tenant_id` (использует tenant-isolation AgenticMind).
- **FR-INTEG-2 (MUST).** Тонкие адаптеры под LangGraph, CrewAI, Claude Agent SDK, OpenAI Agents SDK и raw/manual-режим; всё, что эмитит OTel GenAI спаны, интегрируется в основном маппингом.
- **FR-INTEG-3 (MUST).** Модель-агностичность: провайдер и модель фиксируются атрибутами; APL не привязан к одному провайдеру.
- **FR-INTEG-4 (MUST).** Два режима деплоя: SaaS (хостед AgenticMind) и self-hosted/VPC (enterprise) с одинаковым контрактом.
- **FR-INTEG-5 (SHOULD).** Инструментация как DoD-требование в Agentic Product Standard: SDK встаёт в каждый новый продукт по умолчанию.

## 7. Нефункциональные требования (NFR)
- **NFR-PERF-1 (MUST).** Оверхед инструментации на основной поток продукта < 1% (батч/асинхронный экспорт).
- **NFR-SCALE-1 (MUST).** Хранилище трейсов масштабируется на высокочастотные продукты (ClickHouse); prod-сэмплинг настраиваем.
- **NFR-REL-1 (SHOULD).** Пайплайны APL (ingestion, eval, кластеризация) переживают перезапуск без потери состояния; durable-исполнение для длительных eval/improvement-задач.
- **NFR-SEC-1 (MUST).** Границы прав и разрушительные действия форсируются кодом, а не промптом (Cycle of Trust); авто-улучшение работает только внутри безопасного конверта (см. FR-IMPROVE-4).
- **NFR-SEC-2 (MUST).** Секреты и креды инструментов не хранятся агентом и не попадают в трейсы; scoped-доступ.
- **NFR-PRIV-1 (MUST).** PII редактируется до попадания в хранилище; управляемый content-capture.
- **NFR-PRIV-2 (SHOULD).** Аудируемость: why-trace + audit-лог достаточны для регуляторного разбора (позиционирование под EU AI Act).
- **NFR-TENANT-1 (MUST).** Данные тенантов изолированы; кросс-тенантная утечка невозможна на уровне хранилища и запросов.
- **NFR-PORT-1 (MUST).** Vendor-neutrality: телеметрия читается любым OTel-совместимым бэкендом; отсутствует lock-in формата.
- **NFR-OBS-1 (SHOULD).** Наблюдаемость самого APL (здоровье ingestion, лаги пайплайнов, стоимость судейских прогонов).
- **NFR-DX-1 (MUST).** Time-to-first-trace ≤ 1 день; документация интеграции и адаптеров.

## 8. Модель данных (обязательные сущности)
```
agent            id, product_id, task_description, owner
agent_version    prompt_hash, prompt, tools[], model, params,
                 context_strategy, git_ref, created_at        (иммутабельно)
trace/span       OTel-совместимые; FK -> (agent_id, agent_version)
eval_case        input, reference/rubric, tags, source(curated|mined), agent_id
eval_run         scores, judge_version, agent_version, ts
judge            prompt, calibration(human_labels, TPR, TNR), version
failure          trace_ref, cluster_id, severity, discovered_at
failure_cluster  agent_id, label, description, count, trend, example_refs
annotation       trace_ref, human_label            (калибровка + таксономия)
improvement      version_from, version_to, hypothesis, eval_delta,
                 status(proposed|approved|rejected|deployed), author
scorecard        agent_id -> материализованный вид
```

## 9. Интерфейсы и контракты
- **SDK API (TS/Python):** `wrapAgent(agentId, opts)`, автоинструментация инструментов, `recordOutcome()`, `recordFeedback()`, `recordDecision()`.
- **Ingestion:** OTLP endpoint (совместим с OTel Collector).
- **Export:** OTel-совместимый выход во внешние бэкенды (опция).
- **Improvement webhook/API:** уведомление о предложенном/применённом патче для интеграции с ревью-процессом.
- **Config:** политики сэмплинга, границы автономии, whitelist типов авто-патчей, версия OTel-конвенций.

## 10. Ограничения и допущения
- **C1.** Реализация — модуль AgenticMind (Apache-2.0 core), а не отдельная система.
- **C2.** OTel GenAI conventions на момент разработки в статусе development -> обязательный opt-in для agent/tool-спанов; версия фиксируется.
- **C3.** Стек фиксирован: TS-first SDK, Postgres+pgvector, ClickHouse, OTel Collector.
- **C4.** Tenant-isolation уже присутствует в AgenticMind (v1.4.0) и переиспользуется.
- **A1.** Продукты способны обернуть свой agent loop и пробросить trace context (в т.ч. multi-agent).
- **A2.** Для L3 у агента есть зрелый golden set и откалиброванный судья.

## 11. Критерии приёмки уровня системы (Definition of Done)
- [ ] Трейсы из ≥ 2 продуктов на разных фреймворках дают идентичную структуру дерева и обязательные атрибуты.
- [ ] Multi-agent сценарий отображается одним связным трейсом (нет orphan-спанов).
- [ ] Реестр версий: по трейсу восстанавливается конфиг; тренд score строится по версиям.
- [ ] Golden set на агента; CI-гейт блокирует деплой-регрессию.
- [ ] Каждый активный судья имеет валидную калибровку (TPR/TNR > 80% на ≥ 100 метках).
- [ ] Ошибки агента сведены в именованные кластеры с трендами; алерт на post-deploy всплеск.
- [ ] Цикл L1->L2 работает; L3 доступен как opt-in и не трогает инструменты/права.
- [ ] Скоркарта per-agent + fleet-вид; журнал improvements трассируем.
- [ ] Мультитенантная изоляция подтверждена; PII редактируется до хранилища.
- [ ] Time-to-first-trace ≤ 1 день на новом продукте.

## 12. План поставки по фазам (с гейтами)
- **Фаза 0 — Контракт и реестр.** Заморозить OTel+`moai.*` схему и `agent_version`. *Гейт:* схема утверждена, реестр принимает версии.
- **Фаза 1 — SDK + ingestion + store.** Трейсы текут из одного реального продукта. *Гейт:* time-to-first-trace ≤ 1 день; видимость исполнения.
- **Фаза 2 — Golden set + ассерты + CI-гейт.** *Гейт:* регрессия блокируется автоматически.
- **Фаза 3 — Судьи + калибровка + кластеризация + скоркарта.** *Гейт:* known-where-it-fails по каждому агенту, тренды во времени.
- **Фаза 4 — Цикл L1->L2.** *Гейт:* систематический тюнинг через предложения патчей.
- **Фаза 5 — L3 judge-gated авто + A/B.** *Гейт:* самоулучшение на лучших агентах внутри безопасного конверта.
- **После:** инструментация -> DoD-требование в Agentic Product Standard; SDK по умолчанию в каждом новом продукте.

## 13. Риски и анти-паттерны (как ограничения)
- **R1.** Единый глобальный eval-сьют вместо golden set на агента -> метрика бессмысленна. *Запрещено (FR-EVAL-1).*
- **R2.** Авто-улучшение без откалиброванного судьи -> самоулучшение в сторону шума. *Запрещено (FR-JUDGE-2, FR-IMPROVE-3).*
- **R3.** Авто-патчи, трогающие инструменты/права/границы доверия. *Запрещено (FR-IMPROVE-4).*
- **R4.** Пропуск version-реестра -> нечему атрибутировать прирост/регрессию. *Запрещено (FR-REG).*
- **R5.** Постоянная 100% инструментация на масштабе -> неоправданная стоимость. *Митигируется сэмплингом (FR-SDK-4).*
- **R6.** Vanity-метрика («helpfulness вырос») без трендов по кластерам -> ложное чувство прогресса. *Митигируется FR-FAIL-3.*
- **R7.** Сломанная span lineage через async-границы -> рассыпавшиеся трейсы. *Митигируется FR-CONTRACT-6.*
