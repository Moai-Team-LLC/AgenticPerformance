Q1
RECOMMENDATION: Ship L3 auto-apply for exactly ONE patch class at GA: few-shot exemplars mined from already-passing production traces, appended into a structurally-confined, field-scoped `few_shot[]` slot. Do NOT auto-apply free-form system-prompt rewrites or 'context strategy' changes in v1 — keep those at L2 (human-reviewed). The autonomy boundary must be a mechanical diff-allowlist + content guard, not a category label. Every candidate exemplar (a) must originate from a trace whose outcome=success AND whose grounding-judge verdict was 'supported', (b) must pass the SAME guard.ts injection/PII/exfil detectors that gate agent INPUT plus an added 'is-this-an-instruction-not-an-answer?' judge dimension, (c) must be provenance-tagged with source trace_ref + tenant and quarantined if it came from low-trust/anonymous traffic, and (d) the resulting diff must be rejected if it references any tool name, secret, access verb, or scope language. L3 is opt-in per agent (FR-IMPROVE-6) and only lights up once that agent is L3-eligible (see rationale).
DEFAULT: L3 default patch-type allowlist = { few_shot_append } only. System-prompt edits, tool-description edits, and context-strategy changes default to L2 (suggest-only, human-merge). Enforcement = structured patch_diff confined to the `few_shot[]` field + guard.ts content scan + injection-vs-answer judge on each exemplar + provenance/quarantine tag; auto-merge is un-writable unless the improvement ledger row carries patch_diff, eval_run_id, per-failure-mode delta, judge_id+judge_version+calibration snapshot, canary/A-B result, and source_trace_refs (code-level invariant).
RATIONALE: The PRD (Q1 itself, FR-IMPROVE-4) already leans 'prompt/few-shot yes'. The findings force narrowing further: improve-trust#2 proves the prompt/few-shot boundary is NOT mechanically enforceable as a category ('a prompt is free-form text in the instruction channel'), so a bare 'prompts allowed' default is unsafe. improve-trust#3 shows the existing analogue feedback-promoter.ts already auto-promotes top-scored production answers into knowledge_cards with ONLY a grounding judge and ZERO guard re-run — a live proof that this loop ships poison-able today. Few-shot APPEND (additive, field-scoped) is the only class where a diff-allowlist is trivially checkable and a poisoned example is traceable/purgeable via source_trace_ref; a system-prompt rewrite is an unbounded free-text diff that improve-trust#4 shows is exactly the lever that games an LLM judge. guard.ts already exists (email/phone/card/SSN/IPv4 + EN/RU injection patterns) so the content guard is reuse, not net-new. improve-trust#6 + A2 show L3 preconditions (per-agent golden set + calibrated judge) are among the hardest deliverables in the PRD, so L3 must be a code-enforced eligibility gate that lights up gradually, not a GA-wide switch.
TRADEOFFS: Narrowing to few-shot-append means the highest-leverage fixes (reworking a bad system prompt) stay manual/L2 longer, so autonomous quality gains are slower and some teams will feel L3 is underpowered at launch. The injection-vs-answer judge and provenance/quarantine add per-exemplar LLM cost and a second judge to calibrate. Requiring full ledger fields before any auto-merge raises the bar for shipping L3 at all (some agents never become eligible). The alternative — allowing prompt rewrites — is faster but has no mechanical boundary, shares the judge identity with the optimizer (reward-hackable), and makes a poisoned change durable and hard to attribute; that risk is not acceptable for an EU-AI-Act-positioned product.
